## 1. DeepTwin Vision

### 1.1 Purpose Statement

DeepTwin is the multimodal patient intelligence layer of the DeepSynaps clinical operating system. Its purpose is to synthesize diverse patient data streams into coherent, evidence-linked, uncertainty-aware clinical insights that support — never replace — clinician judgment.

DeepTwin operates as an intelligent observational assistant: it watches patterns across time and data types, flags potential correlations for human attention, and provides the evidentiary scaffolding clinicians need to evaluate those observations in context.

### 1.2 The DeepTwin Philosophy

The architecture is built on five philosophical commitments:

#### 1.2.1 Clinician Partnership, Not Replacement
DeepTwin does not diagnose. It does not prescribe. It does not make treatment decisions. Every output is framed as a hypothesis requiring clinician review — never as a conclusion requiring clinician action.

#### 1.2.2 Epistemic Humility
DeepTwin is explicit about what it does not know. Uncertainty is not an implementation detail to be hidden; it is a first-class feature to be prominently displayed. Every finding carries confidence intervals, evidence grades, and population transferability assessments.

#### 1.2.3 Temporal Groundedness
Clinical reality unfolds in time. DeepTwin respects temporal ordering, recognizes that correlation across time is not causation, and uses temporal analysis methods that preserve — rather than violate — the arrow of time.

#### 1.2.4 Multimodal Integration
The body produces many signals — biochemical, physiological, behavioral, environmental, subjective. DeepTwin treats these as complementary views into patient state, seeking patterns that emerge only when modalities are considered together.

#### 1.2.5 Evidence Linkage
Every hypothesis generated by DeepTwin is tethered to evidence: patient-specific data points, population-level literature, clinical guidelines, or regulatory documents. Hypotheses without evidentiary support are not suppressed — they are labeled accordingly.

### 1.3 What DeepTwin Is NOT

The following boundaries are architectural and non-negotiable:

| Capability | Status | Rationale |
|---|---|---|
| Autonomous diagnosis | **EXPLICITLY EXCLUDED** | Diagnosis requires clinical judgment, physical examination, and contextual interpretation beyond data patterns |
| Autonomous prescribing | **EXPLICITLY EXCLUDED** | Prescribing requires understanding of contraindications, patient preferences, cost considerations, and clinical nuance |
| Causal truth claims | **EXPLICITLY EXCLUDED** | Causality cannot be established from observational data alone; DeepTwin reports correlations with appropriate caveats |
| Definitive prognoses | **EXPLICITLY EXCLUDED** | Outcome predictions are exploratory and framed as probability distributions, not point predictions |
| Substitution for clinical judgment | **EXPLICITLY EXCLUDED** | All outputs require clinician interpretation before any clinical action |

### 1.4 What DeepTwin IS

| Capability | Description | Implementation |
|---|---|---|
| Temporal correlation | Identifies patterns across time and data types | Cross-correlation with lag analysis, DTW alignment |
| Ranked hypotheses | Generates and ranks candidate interpretations | Evidence-weighted scoring, confidence calibration |
| Exploratory synthesis | Combines insights from multiple sources into coherent narratives | Structured generation with citation grounding |
| Uncertainty-aware | Quantifies and displays uncertainty for every output | Bayesian confidence intervals, bootstrap resampling |
| Clinician-reviewed | All outputs flow through clinician review before action | Workflow integration with review/approval gates |
| Evidence-linked | Every hypothesis connected to supporting evidence | Automated literature matching, guideline linkage |

### 1.5 Conceptual Model

DeepTwin operates on a simple conceptual model: **OBSERVE → CORRELATE → HYPOTHESIZE → PRESENT → REVIEW**

### 1.6 Design Principles

**DP-1: Evidence Before Inference**  
No hypothesis is generated without underlying evidence. The evidence chain is always traversable from hypothesis to source data.

**DP-2: Uncertainty as UI**  
Uncertainty is not logged only in the backend; it is prominently displayed in the clinician interface. Users should never need to dig to find confidence metrics.

**DP-3: Temporal Fidelity**  
Time is not treated as a feature vector to be flattened. Temporal ordering, intervals, and sequencing are preserved throughout the pipeline.

**DP-4: Modality Respect**  
Each data type retains its native semantics during analysis. Labs are not flattened into the same representation as vital signs; the correlation engine respects modality-specific properties.

**DP-5: Human-in-the-Loop**  
There is no fully automated path from data to clinical action. A qualified clinician must review and approve every actionable hypothesis.

**DP-6: Safety by Design**  
Safety constraints are architectural, not policy-based. They cannot be bypassed through configuration or API parameters.

### 1.7 Clinical Context

DeepTwin serves multiple clinical contexts:

- **Primary Care:** Longitudinal patient tracking, early pattern detection, preventive care prompts
- **Specialty Care:** Disease-specific monitoring, treatment response tracking, flare detection
- **Hospital Medicine:** Acute deterioration signals, multi-organ dysfunction patterns, discharge readiness
- **Research:** N-of-1 trial support, real-world evidence generation, hypothesis generation for formal studies
- **Population Health:** Cohort identification, risk stratification (exploratory), outcome monitoring

### 1.8 Success Metrics

| Metric | Target | Measurement |
|---|---|---|
| Hypothesis precision | >80% relevant | Clinician rating of hypothesis usefulness |
| Time to insight | <5 min to key patterns | Time from patient selection to prioritized hypotheses |
| False positive rate | <15% clinically irrelevant | Hypotheses dismissed without action |
| Clinician trust score | >4.0/5.0 | Quarterly survey of active users |
| Safety incidents | Zero | Number of autonomous clinical decisions (must be zero) |
| Evidence linkage coverage | >95% | Percentage of hypotheses with evidence links |

## 2. Data Ingestion

### 2.1 Overview

DeepTwin ingests data from 12 distinct sources, each with unique temporal properties, update frequencies, data quality characteristics, and clinical semantics. The ingestion pipeline is designed to normalize these heterogeneous streams into a unified temporal representation while preserving modality-specific metadata.

### 2.2 Data Source Taxonomy

The 12 data sources are organized into four categories:

| Category | Sources | Update Frequency | Clinical Role |
|---|---|---|---|
| **Structured Clinical** | EHR encounters, Lab results, Medication records | Per-encounter, batch, event-driven | Objective clinical measures |
| **Continuous Physiologic** | Vital signs (monitor), Wearable devices | Continuous / 1-60 min | Real-time physiological state |
| **Molecular** | Genomics, Pharmacogenomics | Once / per-test | Biological predisposition |
| **Subjective & Environmental** | Patient-reported outcomes, Environmental, Social determinants | Daily / periodic | Patient experience and context |

### 2.3 Source 1: EHR Encounter Data

**Description:** Clinical notes, diagnoses (ICD-10), procedures (CPT), problem lists, encounter metadata from the electronic health record.

**Ingestion Pipeline:**

```
HL7 FHIR R4 Resources:
  - Encounter (encounter dates, type, location, provider)
  - Condition (problems, diagnoses, onset/abatement dates)
  - Procedure (performed procedures, dates, outcomes)
  - Observation (clinical observations, interpretations)

Ingestion Steps:
  1. FHIR resource retrieval via R4 API
  2. Patient ID resolution (MPI match)
  3. De-duplication (encounter-level)
  4. Temporal extraction (all date/time fields normalized to ISO 8601)
  5. Coding standardization (ICD-10-CM, CPT-4, SNOMED CT mapping)
  6. Text extraction for NLP pipeline (clinical notes)
  7. Quality scoring (completeness, timeliness, consistency)
  8. Feature extraction (condition count, encounter frequency, provider count)
  9. Temporal alignment (bucketed to observation timeline)

Normalization:
  - All dates → UTC timestamps with timezone preservation
  - All codes → standard coding systems with crosswalks
  - All providers → provider directory IDs
  - All locations → facility registry IDs

Feature Extraction:
  - Encounter frequency (rolling 30/90/365 day counts)
  - Active condition count
  - Condition onset velocity (new diagnoses per period)
  - Provider continuity index
  - Care setting transitions (inpatient → outpatient → ED)

Temporal Alignment:
  - Primary: Encounter date
  - Secondary: Condition onset dates, procedure dates
  - Resolution: Where available, condition abatement dates

Missing Data Handling:
  - Missing encounter notes → flag as incomplete, use structured data only
  - Missing discharge summaries → request from source system
  - Coding gaps → NLP backfill for uncoded encounters
  - Historical data gaps → annotate coverage periods

Quality Scoring:
  - Completeness: % of expected fields populated (target >90%)
  - Timeliness: Lag between event and documentation (target <24h)
  - Consistency: Internal coherence of diagnosis-procedure pairs
  - Plausibility: Temporal ordering constraints (no future dates)
```

### 2.4 Source 2: Laboratory Results

**Description:** Quantitative and qualitative laboratory test results from hospital labs, reference labs, and point-of-care testing.

**Ingestion Pipeline:**

```
HL7 FHIR Resources:
  - Observation (lab results with reference ranges, interpretations)
  - DiagnosticReport (report-level organization)
  - Specimen (collection, handling metadata)

Ingestion Steps:
  1. FHIR Observation retrieval
  2. LOINC code normalization (LOINC 2.74+)
  3. Unit standardization (UCUM mapping)
  4. Reference range contextualization (age, sex, pregnancy-specific)
  5. Flag interpretation (normal/low/high/critical)
  6. Delta calculation (change from prior result)
  7. Temporal ordering (collection time > received time > result time)
  8. Quality scoring
  9. Feature extraction
  10. Temporal alignment

Normalization:
  - All results → standard units via UCUM conversion
  - LOINC codes → latest version with mapping history
  - Reference ranges → population-specific where available
  - Non-numeric results → coded values (positive/negative, detected/not detected)

Feature Extraction:
  - Result value (numeric, normalized to standard units)
  - Z-score relative to reference range
  - Percentile within population distribution
  - Delta from prior (absolute and percentage)
  - Delta velocity (rate of change)
  - Abnormality flags (binary, count)
  - Test frequency per analyte
  - Panel completeness

Temporal Alignment:
  - Primary: Specimen collection datetime
  - Secondary: Result availability datetime
  - Tertiary: Order datetime

Missing Data Handling:
  - Ordered but not resulted → pending queue with SLA monitoring
  - Resulted but not transmitted → reconciliation job every 4 hours
  - Historical reference ranges → use current ranges with annotation
  - Missing collection time → use receipt time with uncertainty flag

Quality Scoring:
  - Pre-analytic: Specimen quality flags, collection-to-receipt time
  - Analytic: Delta check violations, critical value confirmation
  - Post-analytic: Turnaround time, transmission completeness
  - Longitudinal: Result plausibility given patient history
```

### 2.5 Source 3: Medication Records

**Description:** Prescribed, dispensed, and administered medications including dosing, timing, adherence indicators, and medication history.

**Ingestion Pipeline:**

```
HL7 FHIR Resources:
  - MedicationRequest (prescriptions, orders)
  - MedicationDispense (pharmacy fulfillment)
  - MedicationAdministration (inpatient administration)
  - MedicationStatement (patient-reported medications)

Ingestion Steps:
  1. FHIR medication resource retrieval
  2. RxNorm normalization (ingredient + strength + form)
  3. ATC classification (5-level hierarchy)
  4. Dose standardization (to common units)
  5. Schedule parsing (SIG parsing for frequency/duration)
  6. Adherence estimation (pharmacy refill intervals)
  7. Drug-drug interaction pre-screening
  8. Quality scoring
  9. Feature extraction
  10. Temporal alignment

Normalization:
  - All medications → RxNorm normalized names
  - All doses → mg equivalents where possible
  - All schedules → structured frequency + duration
  - Route of administration → standard terminology

Feature Extraction:
  - Active medication count (by ATC class)
  - Medication burden indices (DDS, medication count)
  - Adherence ratio (possession ratio, MPR, PDC)
  - Dose trajectories (titration patterns)
  - Switch patterns (medication changes within class)
  - Gap analysis (unfilled prescriptions)
  - High-risk medication flags (anticoagulants, opioids, hypoglycemics)

Temporal Alignment:
  - Primary: Prescription start date
  - Secondary: Dispense dates (actual pharmacy fulfillment)
  - Tertiary: Administration timestamps (inpatient)

Missing Data Handling:
  - OTC medications → patient-reported, lower confidence tier
  - Historical medications → reconcile with pharmacy claims
  - PRN administrations → frequency estimation from actual data
  - Non-adherence → annotated in adherence modeling

Quality Scoring:
  - Completeness: % of prescriptions with full SIG, dose, duration
  - Timeliness: Pharmacy fill within 48h of prescription
  - Consistency: Ordered vs. dispensed vs. administered reconciliation
  - Accuracy: Pharmacy verification of medication identity
```

### 2.6 Source 4: Vital Signs (Continuous Monitoring)

**Description:** High-frequency physiological measurements from bedside monitors, telemetry, and continuous monitoring devices.

**Ingestion Pipeline:**

```
HL7 FHIR Resources:
  - Observation (vital signs with device provenance)
  - Device (monitor metadata)
  - DeviceMetric (measurement specifications)

Ingestion Steps:
  1. Streaming data ingestion (HL7 v2 ORU messages or FHIR Subscriptions)
  2. Device identification and validation
  3. Signal quality assessment (artifact detection)
  4. Physiologic plausibility checking
  5. Gap detection and interpolation annotation
  6. Aggregation (raw → 1-min → 5-min → hourly windows)
  7. Quality scoring
  8. Feature extraction
  9. Temporal alignment

Normalization:
  - All units → standard clinical units (°C, mmHg, bpm, %, breaths/min)
  - Device-specific calibration offsets → documented
  - Measurement site → standardized (e.g., BP: right arm, supine)

Feature Extraction:
  - Central tendency: mean, median per window
  - Variability: SD, CV, range per window
  - Extrema: min, max per window
  - Trend: slope, direction per window
  - Episode detection: hypotension, hypertension, tachycardia, bradycardia
  - Cross-parameter indices: shock index, pulse pressure
  - Variability metrics: heart rate variability (SDNN, RMSSD)

Temporal Alignment:
  - Primary: Measurement timestamp (device-generated)
  - Resolution: 1-minute buckets for continuous, event-time for episodic

Missing Data Handling:
  - Signal dropouts <5 min → linear interpolation with flag
  - Signal dropouts 5-30 min → carry-forward with uncertainty
  - Signal dropouts >30 min → null with coverage gap annotation
  - Artifact periods → nullify with artifact flag

Quality Scoring:
  - Signal quality index (SQI): 0-100 per measurement
  - Artifact burden: % of time with artifacts
  - Coverage: % of expected measurements received
  - Latency: time from measurement to system availability
```

### 2.7 Source 5: Wearable Devices

**Description:** Consumer and medical-grade wearable data including activity, sleep, heart rate, blood oxygen, ECG, temperature, and other biometrics.

**Ingestion Pipeline:**

```
Integration Methods:
  - Apple HealthKit / Google Health Connect (patient-authorized)
  - Device-specific APIs (Fitbit, Garmin, Oura, etc.)
  - Medical device integrations (Zio patch, AliveCor, etc.)

Ingestion Steps:
  1. Patient-authorized data connection
  2. OAuth token management with refresh
  3. Data pull (incremental, daily batch)
  4. Schema mapping (device-specific → standard schema)
  5. Unit conversion and validation
  6. Data quality assessment (consumer-grade limitations)
  7. Feature extraction
  8. Temporal alignment

Normalization:
  - Activity: steps, distance, floors, active minutes → standard units
  - Sleep: duration, stages, efficiency → standard classification
  - Heart rate: bpm with quality indicators
  - HRV: time-domain and frequency-domain metrics
  - SpO2: percentage with measurement confidence
  - Temperature: °C with measurement site annotation
  - ECG: classification labels (sinus rhythm, AFib, etc.)

Feature Extraction:
  - Activity: daily steps, active minutes, sedentary time, intensity distribution
  - Sleep: total time, latency, efficiency, stage distribution, wake episodes
  - Cardiovascular: resting HR, HR range, HRV trends, arrhythmia burden
  - Respiratory: SpO2 trends, estimated RR from accelerometer
  - Circadian: rhythm stability, chronotype estimation

Temporal Alignment:
  - Primary: Device-reported timestamp
  - Resolution: Varies by data type (1 min for HR, daily for sleep, per-event for ECG)

Missing Data Handling:
  - Device not worn → null with compliance annotation
  - Sync gaps → last-known-value with staleness flag
  - Battery death → coverage gap with expected duration
  - Consumer-grade limitations → lower evidence tier annotation

Quality Scoring:
  - Wear compliance: % of expected wearing time
  - Data completeness: % of expected measurements
  - Signal quality: device-reported confidence where available
  - Consumer vs. medical grade: evidence tier classification
```

### 2.8 Source 6: Genomic Data

**Description:** Germline genetic variants, somatic mutations, pharmacogenomic results, and polygenic risk scores.

**Ingestion Pipeline:**

```
Data Formats:
  - VCF (Variant Call Format) 4.2+
  - Pharmacogenomic reports (CPIC-aligned star alleles)
  - Microarray genotype data
  - Sequencing reports (germline and somatic)

Ingestion Steps:
  1. VCF parsing and validation
  2. Variant normalization (left-alignment, decomposition)
  3. GRCh37/GRCh38 liftover if needed
  4. Annotation (VEP, ClinVar, PharmGKB)
  5. Pharmacogenomic phenotype inference (CPIC guidelines)
  6. Pathogenicity scoring (ACMG criteria)
  7. Quality scoring
  8. Feature extraction
  9. Temporal alignment (one-time with update capability)

Normalization:
  - All variants → HGVS nomenclature
  - All genes → HGNC approved symbols
  - Star alleles → CPIC standard nomenclature
  - Phenotypes → inferred metabolizer status (for pharmacogenes)

Feature Extraction:
  - Pharmacogenomic phenotypes: metabolizer status per gene
  - Pathogenic variants: ACMG classification, condition associations
  - Polygenic risk scores: condition-specific PRS where available
  - Carrier status: autosomal recessive carrier states
  - Pharmacokinetic implications: enzyme/transporter variants
  - Pharmacodynamic implications: target/receptor variants

Temporal Alignment:
  - Static data with effective date (test date)
  - Update capability if re-testing occurs

Missing Data Handling:
  - No-call variants → annotate as uncertain
  - Failed regions → coverage gap annotation
  - Ambiguous star alleles → most likely with uncertainty flag
  - Population-specific limitations → ancestry-contextualized interpretation

Quality Scoring:
  - Sequencing depth: mean coverage, % at >20x
  - Genotype quality: Phred-scaled quality scores
  - Concordance: with prior testing if available
  - Coverage completeness: % of target regions with adequate coverage
```

### 2.9 Source 7: Pharmacogenomic Results

**Description:** CPIC guideline-aligned pharmacogenomic test results linking genetic variants to drug metabolism phenotypes.

**Ingestion Pipeline:**

```
Data Sources:
  - Laboratory pharmacogenomic reports
  - CPIC guideline phenotype tables
  - PharmGKB clinical annotations
  - FDA pharmacogenomic biomarker table

Ingestion Steps:
  1. Report parsing (structured + unstructured)
  2. Star allele extraction and validation
  3. CPIC phenotype lookup (e.g., CYP2D6 *1/*4 → intermediate metabolizer)
  4. Drug-gene pair identification (which drugs affected)
  5. Clinical annotation retrieval (guideline recommendations)
  6. Quality scoring
  7. Feature extraction
  8. Temporal alignment

Normalization:
  - Genes → HGNC symbols (CYP2D6, CYP2C19, CYP3A4, SLCO1B1, etc.)
  - Phenotypes → CPIC standard terms (poor, intermediate, normal, rapid, ultrarapid)
  - Drugs → RxNorm normalized with pharmacogenomic implications

Feature Extraction:
  - Metabolizer status per pharmacogene (CYP family, TPMT, DPYD, etc.)
  - CPIC guideline level (1A, 1B, 2A — strong, moderate)
  - Affected medication list (current + potential future)
  - Dosing implication category (standard, reduced, increased, avoid)
  - Alternative medication suggestions per CPIC

Quality Scoring:
  - Genotype confidence: call quality, coverage
  - Phenotype assignment: certainty of metabolizer classification
  - Clinical relevance: CPIC guideline strength
  - Recency: guideline version currency
```

### 2.10 Source 8: Patient-Reported Outcomes (PROs)

**Description:** Structured surveys, symptom diaries, quality of life measures, and functional assessments reported by patients.

**Ingestion Pipeline:**

```
Data Collection:
  - PROMIS instruments (CAT and fixed-form)
  - Condition-specific questionnaires (PHQ-9, GAD-7, etc.)
  - Custom symptom diaries
  - Quality of life measures (EQ-5D, SF-36)

Ingestion Steps:
  1. Survey delivery (mobile app, portal, SMS)
  2. Response collection with timestamp
  3. Validation (completeness, logic checks, range validation)
  4. Score computation (raw → T-scores where applicable)
  5. Normative comparison (population-based percentiles)
  6. Quality scoring
  7. Feature extraction
  8. Temporal alignment

Normalization:
  - All scores → T-scores where PROMIS norms available
  - All responses → coded values with reverse-scoring applied
  - Missing items → prorated scores where guidelines allow

Feature Extraction:
  - Total scores (per instrument)
  - Subscale scores (where applicable)
  - Change scores (from prior assessment)
  - Response trajectories (improving/stable/worsening)
  - Threshold crossings (mild/moderate/severe)
  - Response latency (time to complete — engagement indicator)

Temporal Alignment:
  - Primary: Response timestamp
  - Resolution: Per-assessment (typically daily to weekly)

Missing Data Handling:
  - Missed assessments → compliance tracking with gentle reminders
  - Partial completion → prorated scoring where valid
  - Straight-lining → flag as low-quality response
  - Response fatigue → completion time analysis

Quality Scoring:
  - Completion rate: % of scheduled assessments completed
  - Response quality: consistency checks, pattern detection
  - Timeliness: response within expected window
  - Engagement trend: completion rate trajectory
```

### 2.14 Source 12: Claims and Cost Data

**Description:** Healthcare utilization claims, cost data, care gaps, and quality measure gaps from payers and practice management systems.

**Ingestion Pipeline:**

```
Data Sources:
  - HL7 FHIR ExplanationOfBenefit
  - Practice management system exports
```

### 2.16 Data Quality Framework

Every ingested data point receives a composite quality score:

```
Q_composite = w1 * Q_completeness + w2 * Q_timeliness + w3 * Q_consistency + w4 * Q_plausibility

Where:
  Q_completeness  = % of expected fields populated
  Q_timeliness    = recency of data relative to event time
  Q_consistency   = internal coherence with other data from same source
  Q_plausibility  = physiological/clinical reasonableness
  
  w1 + w2 + w3 + w4 = 1.0 (configurable weights, defaults to 0.25 each)
```

Quality scores propagate through the pipeline and influence:
- Hypothesis confidence weighting (lower quality data = wider confidence intervals)
- Evidence grade assignment (data quality bounds evidence quality)
- Alert prioritization (higher quality data prioritized in hypothesis ranking)
- Display prominence (quality indicators shown in clinician interface)

## 3. Multimodal Correlation Engine

### 3.1 Overview

The Multimodal Correlation Engine is the analytical core of DeepTwin. It identifies statistical relationships across the 12 data sources, accounting for temporal dynamics, variable measurement scales, and the irregular observation patterns inherent in clinical data.

The engine operates on the principle that clinically meaningful patterns often manifest across multiple data types — a medication change may correlate with lab shifts, symptom changes, and activity level changes, but each at different temporal lags and scales. The correlation engine detects these cross-modality relationships while maintaining appropriate epistemic humility.

### 3.2 Architectural Principles

**MP-1: Exploratory, Not Confirmatory**  
All correlation analyses are exploratory. The engine generates candidate relationships for human evaluation; it does not validate causal claims.

**MP-2: Temporal Respect**  
Correlation is computed with temporal ordering constraints. Future events cannot cause past events; the engine enforces this through directed analysis.

**MP-3: Scale Awareness**  
Different data types have different natural timescales. The engine adapts analysis windows to each modality pair rather than forcing uniform windows.

**MP-4: Uncertainty Propagation**  
Correlation strength estimates include confidence intervals derived from bootstrap resampling. Point estimates without uncertainty are not reported.

**MP-5: Multiple Method Convergence**  
No single statistical method is relied upon. The same modality pair is analyzed with multiple complementary techniques, and convergence across methods increases confidence.

### 3.4 Correlation Methods Portfolio

#### 3.4.1 Pearson and Spearman Correlation

**When to use:** Linear and monotonic relationships between numeric variables (e.g., lab value vs. vital sign).

```python
# Conceptual implementation
for each modality_pair (m1, m2):
    aligned_data = temporal_align(m1, m2, max_lag=LAG_MAX)
    
    # Pearson (linear)
    r_pearson, p_pearson = pearsonr(aligned_data.m1, aligned_data.m2)
    
    # Spearman (monotonic, rank-based)
    r_spearman, p_spearman = spearmanr(aligned_data.m1, aligned_data.m2)
    
    # Bootstrap confidence intervals
    ci_pearson = bootstrap_ci(aligned_data, method=pearsonr, n_bootstrap=1000)
    ci_spearman = bootstrap_ci(aligned_data, method=spearmanr, n_bootstrap=1000)
    
    store_correlation_result(
        modalities=(m1, m2),
        pearson=(r_pearson, ci_pearson, p_pearson),
        spearman=(r_spearman, ci_spearman, p_spearman),
        sample_size=len(aligned_data),
        time_window=(aligned_data.start, aligned_data.end)
    )
```

**Parameters:**
- `LAG_MAX`: Maximum temporal lag to consider (default 30 days for labs-meds, 7 days for vitals-meds, 90 days for EHR-labs)
- `TEMPORAL_CONSTRAINT`: Directional — m1 must precede m2 by at least 0 time units (no reverse causation)
- `MIN_SAMPLES`: Minimum 10 paired observations for reporting

**Evidence tier:** Correlational — not causal. Flagged as "temporal association detected."

#### 3.4.2 Mutual Information

**When to use:** Non-linear relationships, mixed variable types (numeric vs. categorical), complex interactions that may not be captured by linear correlation.

```python
# Conceptual implementation
for each modality_pair (m1, m2):
    # Discretize continuous variables using adaptive binning
    m1_discrete = adaptive_discretize(m1, method='kmeans', max_bins=20)
    m2_discrete = adaptive_discretize(m2, method='kmeans', max_bins=20)
    
    # Compute mutual information
    mi = mutual_information(m1_discrete, m2_discrete, method='knn', k=5)
    
    # Normalize to [0,1] using min-max normalization
    mi_normalized = mi / min(H(m1), H(m2))  # normalized mutual information
    
    # Statistical significance via permutation test
    p_value = permutation_test_mi(m1, m2, n_permutations=1000)
    
    # Bootstrap confidence interval
    ci_mi = bootstrap_ci((m1_discrete, m2_discrete), method=mutual_information)
    
    store_mi_result(
        modalities=(m1, m2),
        mutual_information=(mi, ci_mi, p_value),
        normalized_mi=mi_normalized,
        sample_size=len(m1),
        non_linear_indicator=(mi_normalized > r_pearson**2)  # MI > R² suggests non-linearity
    )
```

**Evidence tier:** Associational — captures any statistical dependency, including non-linear.

#### 3.4.3 Temporal Lag Analysis

**When to use:** Determining optimal lag between events in different modalities (e.g., medication start → lab change latency).

```python
# Conceptual implementation
for each modality_pair (m1, m2) where temporal_direction(m1 → m2):
    lag_results = []
    
    for lag in LAG_RANGE:  # e.g., 0 to 90 days in 1-day increments
        # Shift m1 forward by lag, compute correlation
        m1_shifted = shift_temporal(m1, lag=lag)
        aligned = intersect_temporal(m1_shifted, m2)
        
        if len(aligned) >= MIN_SAMPLES:
            r, p = pearsonr_or_spearmanr(aligned.m1, aligned.m2)
            lag_results.append({
                'lag': lag,
                'correlation': r,
                'p_value': p,
                'n_samples': len(aligned)
            })
    
    # Find optimal lag
    optimal_lag = max(lag_results, key=lambda x: abs(x['correlation']) if x['p_value'] < 0.05 else 0)
    
    # Lag confidence via bootstrap
    ci_optimal_lag = bootstrap_lag_ci(m1, m2, lag_range=LAG_RANGE)
    
    store_lag_result(
        modalities=(m1, m2),
        optimal_lag_days=optimal_lag['lag'],
        lag_confidence_interval=ci_optimal_lag,
        correlation_at_optimal_lag=optimal_lag['correlation'],
        lag_landscape=lag_results  # Full lag-correlation curve for visualization
    )
```

**Parameters:**
- `LAG_RANGE`: Modality-specific (labs-meds: 0-30 days; vitals-meds: 0-7 days; PROs-meds: 0-14 days)
- `LAG_STEP`: Daily resolution
- `MIN_SAMPLES`: 10 observations minimum at any lag

**Evidence tier:** Temporal associational — identifies time-delayed relationships.

#### 3.4.4 Granger Causality (Exploratory Only)

**When to use:** Assessing whether one time series provides predictive information about another, with strong caveats about causal interpretation.

```python
# Conceptual implementation — with extensive safety framing
for each modality_pair (m1, m2):
    # Prerequisites check
    if not is_stationary(m1) or not is_stationary(m2):
        m1_diff = difference_to_stationarity(m1)
        m2_diff = difference_to_stationarity(m2)
    
    # Test: Does m1 Granger-cause m2?
    f_stat, p_value, lag_order = granger_causality_test(
        cause=m1, effect=m2,
        max_lag=MAX_LAG,
        test='ssr_ftest'
    )
    
    # Model comparison: VAR with vs. without m1
    aic_with = var_model([m1, m2], include_m1=True).aic
    aic_without = var_model([m2], include_m1=False).aic
    
    # Safety framing
    safety_label = classify_granger_safety(
        p_value=p_value,
        effect_size=f_stat,
        sample_size=len(m1),
        stationarity_method='ADF test',
        model_fit=aic_comparison
    )
    
    store_granger_result(
        modalities=(m1, m2),
        f_statistic=f_stat,
        p_value=p_value,
        optimal_lag=lag_order,
        effect_size=compute_effect_size(m1, m2),
        # EXPLICIT SAFETY FRAMING
        interpretation='EXPLORATORY: Granger causality tests predictive precedence, NOT causal mechanism',
        required_caveats=[
            'Unobserved confounders may explain observed association',
            'Results sensitive to model specification and lag choice',
            'Stationarity transformation may alter interpretability',
            'Cannot rule out reverse causation or common cause'
        ],
        evidence_grade=map_to_evidence_grade(p_value, sample_size, robustness_checks),
        safety_label=safety_label  # Must be 'EXPLORATORY' — never 'CONFIRMATORY'
    )
```

**CRITICAL SAFETY CONSTRAINTS:**
1. Results are **always** labeled as "exploratory temporal precedence analysis" — never "causality"
2. Granger results are **never** used as sole evidence for any hypothesis
3. A minimum of 100 time points required (higher than typical to reduce spurious results)
4. Model diagnostics (residual autocorrelation, heteroskedasticity) must pass
5. Univariate stationarity confirmed via ADF test (p < 0.05)
6. Results suppressed if VAR model stability conditions not met

**Evidence tier:** EXPLORATORY ONLY — labeled as "predictive precedence detected, causal interpretation not warranted"

#### 3.4.5 Canonical Correlation Analysis (CCA)

**When to use:** Finding linear relationships between two **sets** of variables (e.g., a panel of labs vs. a set of vital signs).

```python
# Conceptual implementation
for each modality_group_pair (group_A, group_B):
    # group_A: e.g., [CRP, WBC, ESR, ferritin] — inflammatory labs
    # group_B: e.g., [HR, temp, BP_variability] — physiologic indicators
    
    # Standardize variables
    A_std = standardize(group_A)
    B_std = standardize(group_B)
    
    # CCA
    cca = CanonicalCorrelationAnalysis()
    cca.fit(A_std, B_std)
    
    # Canonical correlations
    canonical_correlations = cca.canonical_correlations_
    
    # Wilks' lambda test for significance
    for i, rho in enumerate(canonical_correlations):
        wilks_lambda, p_value = wilks_lambda_test(cca, dimension=i)
        
        # Canonical loadings (variable importance)
        loadings_A = cca.x_loadings_[:, i]
        loadings_B = cca.y_loadings_[:, i]
        
        # Redundancy coefficients
        redundancy_A = compute_redundancy(cca, A_std, dimension=i)
        redundancy_B = compute_redundancy(cca, B_std, dimension=i)
        
        store_cca_result(
            dimensions=(group_A, group_B),
            canonical_correlation=rho,
            p_value=p_value,
            loadings_A=dict(zip(group_A.columns, loadings_A)),
            loadings_B=dict(zip(group_B.columns, loadings_B)),
            redundancy_A=redundancy_A,
            redundancy_B=redundancy_B,
            interpretation=f'Multivariate relationship between {len(group_A)} variables and {len(group_B)} variables'
        )
```

**Evidence tier:** Multivariate associational — identifies composite patterns across variable groups.

#### 3.4.6 Dynamic Time Warping (DTW) for Alignment

**When to use:** Finding optimal alignment between two temporal sequences that may have different speeds or timings (e.g., medication onset → symptom change with variable latency).

```python
# Conceptual implementation
for each temporal_sequence_pair (seq_A, seq_B):
    # DTW with Sakoe-Chiba band constraint
    dtw_distance, warp_path = dynamic_time_warping(
        seq_A, seq_B,
        distance_metric='euclidean',
        constraint='sakoe_chiba',
        band_width=max(len(seq_A), len(seq_B)) * 0.2  # 20% window
    )
    
    # Normalize DTW distance by path length
    normalized_dtw = dtw_distance / len(warp_path)
    
    # Warp path analysis — where do sequences align?
    alignment_segments = analyze_warp_path(warp_path)
    # e.g., "seq_A[10:20] aligns with seq_B[15:30] with compression factor 0.67"
    
    # Statistical significance via permutation
    p_value = dtw_permutation_test(seq_A, seq_B, n_permutations=500)
    
    # Bootstrap confidence interval for DTW distance
    ci_dtw = bootstrap_ci((seq_A, seq_B), method=dynamic_time_warping)
    
    store_dtw_result(
        sequences=(seq_A, seq_B),
        dtw_distance=(normalized_dtw, ci_dtw),
        warp_path=warp_path,
        alignment_segments=alignment_segments,
        p_value=p_value,
        interpretation='Optimal temporal alignment identified — similar shape patterns with variable timing'
    )
```

**Parameters:**
- `constraint`: Sakoe-Chiba band to prevent pathological alignments
- `band_width`: 20% of sequence length (prevents excessive warping)
- `distance_metric`: Euclidean for numeric, custom for mixed types

**Evidence tier:** Shape associational — identifies similar temporal patterns regardless of absolute timing.

### 3.6 Method Convergence Framework

When multiple methods agree, confidence increases. When they disagree, the system flags uncertainty.

| Agreement Pattern | Confidence Adjustment | Action |
|---|---|---|
| Pearson + Spearman + MI agree (strong, significant) | +2 levels | Higher evidence grade |
| Pearson + Spearman agree, MI differs | +1 level, flag non-linearity | Note possible non-linear relationship |
| Pearson + MI agree, Spearman differs | 0 adjustment, flag outlier sensitivity | Suggest outlier influence |
| All methods disagree | -2 levels | Flag as unreliable, do not generate hypothesis |
| Granger significant, others not | No bonus — Granger standalone is weak | Lower weight on Granger finding |
| DTW low distance, correlation methods weak | 0 adjustment | Note similar shapes but weak linear assoc |
| CCA significant, bivariate methods weak | +1 level (multivariate pattern) | Highlight as composite pattern |

### 4.3 Anomaly Detection

#### 4.3.1 Univariate Anomaly Detection

```python
# Per-modality, per-variable anomaly detection
for each (modality, variable) in patient_timeline:
    # Method 1: Statistical outlier (IQR-based)
    q1, q3 = variable.quantile([0.25, 0.75])
    iqr = q3 - q1
    outliers_iqr = variable[(variable < q1 - 1.5*iqr) | (variable > q3 + 1.5*iqr)]
    
    # Method 2: Z-score (patient-specific baseline)
    patient_mean = variable.rolling(window='90D').mean()
    patient_std = variable.rolling(window='90D').std()
    z_scores = (variable - patient_mean) / patient_std
    outliers_zscore = variable[abs(z_scores) > 3]
    
    # Method 3: Isolation Forest (multivariate context)
    iso_forest = IsolationForest(contamination=0.05)
    context_features = extract_context_features(variable, patient_timeline)
    anomaly_labels = iso_forest.fit_predict(context_features)
    outliers_ml = variable[anomaly_labels == -1]
    
    # Method 4: Patient-specific deviation (most important)
    patient_baseline = compute_patient_baseline(variable, window='6M')
    deviation = (variable - patient_baseline.mean) / patient_baseline.std
    patient_specific_anomalies = variable[abs(deviation) > 2]
    
    # Consensus anomaly flagging
    anomalies = consensus_vote([outliers_iqr, outliers_zscore, outliers_ml, patient_specific_anomalies])
```

### 4.7 Hypothesis Ranking

#### 4.7.1 Ranking Algorithm

```python
def rank_hypotheses(hypotheses, patient_context):
    """Rank hypotheses by composite score."""
    
    for hypothesis in hypotheses:
        # Evidence strength component (30%)
        evidence_score = (
            0.4 * hypothesis.evidence_grade_score +  # A=1.0, B=0.75, C=0.5, D=0.25
            0.3 * hypothesis.correlation_strength +
            0.2 * hypothesis.sample_size_score +  # Scaled by n
            0.1 * hypothesis.method_convergence_score  # Cross-method agreement
        )
        
        # Clinical relevance component (30%)
        relevance_score = (
            0.4 * hypothesis.clinical_significance +
            0.3 * hypothesis.safety_sensitivity +  # Higher for safety-related
            0.2 * hypothesis.urgency +
            0.1 * hypothesis.actionability  # Can something be done?
        )
        
        # Uncertainty component (20%) — INVERSE, more uncertainty = lower score
        uncertainty_score = 1.0 - hypothesis.combined_uncertainty
        
        # Temporal recency component (10%)
        recency_score = temporal_decay(
            hypothesis.central_timestamp,
            half_life='30D',
            max_age='180D'
        )
        
        # Source reliability component (10%)
        reliability_score = (
            0.5 * hypothesis.data_quality_score +
            0.5 * hypothesis.source_reliability  # Labs > Wearables, etc.
        )
        
        # Composite ranking score
        hypothesis.rank_score = (
            0.30 * evidence_score +
            0.30 * relevance_score +
            0.20 * uncertainty_score +
            0.10 * recency_score +
            0.10 * reliability_score
        )
    
    return sorted(hypotheses, key=lambda h: h.rank_score, reverse=True)
```

## 5. Uncertainty Quantification

### 5.1 Philosophy of Uncertainty

DeepTwin treats uncertainty not as a statistical inconvenience but as a core clinical communication. Every number presented to a clinician carries uncertainty because clinical reality is inherently uncertain. The goal is not to eliminate uncertainty but to measure it, communicate it, and enable clinicians to factor it into their decisions.

**Principles:**
- UQ-1: Uncertainty is a first-class citizen, not metadata
- UQ-2: Uncertainty must be visible in the UI, not hidden in tooltips
- UQ-3: Uncertainty influences hypothesis ranking (higher uncertainty = lower rank)
- UQ-4: Uncertainty is honest — never understated, never overstated
- UQ-5: Uncertainty is contextual — a wide CI for a novel biomarker is different from a wide CI for creatinine

### 5.2 Confidence Intervals

Every quantitative finding in DeepTwin includes a confidence interval computed via one or more methods:

#### 5.2.1 Bootstrap Confidence Intervals

```python
def compute_bootstrap_ci(data, statistic_func, n_bootstrap=2000, ci_level=0.95):
    """Compute bootstrap confidence interval for any statistic."""
    
    bootstrap_statistics = []
    n = len(data)
    
    for _ in range(n_bootstrap):
        # Resample with replacement
        resample = np.random.choice(data, size=n, replace=True)
        stat = statistic_func(resample)
        bootstrap_statistics.append(stat)
    
    # BCa (bias-corrected and accelerated) CI — more accurate than percentile
    ci_lower, ci_upper = compute_bca_interval(
        bootstrap_statistics,
        original_statistic=statistic_func(data),
        data=data
    )
    
    return {
        'point_estimate': statistic_func(data),
        'ci_lower': ci_lower,
        'ci_upper': ci_upper,
        'ci_width': ci_upper - ci_lower,
        'n_bootstrap': n_bootstrap,
        'method': 'BCa bootstrap'
    }
```

### 5.3 Evidence Grading System

DeepTwin uses a 4-tier evidence grading system adapted from GRADE:

#### 5.3.1 Grade A — High Confidence

**Criteria:**
- Large effect size (r > 0.5 or equivalent)
- Large sample (n > 100 paired observations)
- Consistent across multiple analysis methods
- Consistent with established clinical literature
- High data quality (>90% completeness)
- Tight confidence intervals

**Display:** Green badge "Strong Evidence"  
**Action:** Clinician can have high confidence in the finding  
**Example:** Statin therapy → LDL reduction in patient with normal CYP3A4 metabolism

#### 5.3.2 Grade B — Moderate Confidence

**Criteria:**
- Moderate effect size (r = 0.3-0.5)
- Moderate sample (n = 30-100)
- Generally consistent across methods with minor discrepancies
- Supported by some literature but not universally
- Good data quality (75-90% completeness)
- Moderate confidence intervals

**Display:** Blue badge "Moderate Evidence"  
**Action:** Reasonable confidence but clinical judgment essential  
**Example:** Activity level increase → blood pressure improvement in hypertensive patient

#### 5.3.3 Grade C — Low Confidence

**Criteria:**
- Small effect size (r = 0.1-0.3)
- Small sample (n = 10-30)
- Inconsistent across methods
- Limited or indirect literature support
- Moderate data quality (50-75% completeness)
- Wide confidence intervals

**Display:** Yellow badge "Limited Evidence"  
**Action:** Consider as exploratory only; do not base clinical decisions solely on this  
**Example:** Environmental pollen increase → respiratory symptom changes (only PROs and vitals, small n)

#### 5.3.4 Grade D — Very Low Confidence

**Criteria:**
- Very small or uncertain effect size (r < 0.1 or CI includes zero for directional claims)
- Very small sample (n < 10)
- Methods disagree substantially
- No direct literature support
- Poor data quality (<50% completeness)
- Very wide confidence intervals

**Display:** Red badge "Insufficient Evidence"  
**Action:** Not suitable for clinical decision-making; may indicate need for more data  
**Example:** Genetic variant association with rare symptom based on n=3 observations

### 5.4 Sample Size Considerations

```python
def assess_sample_size_adequacy(n, effect_size, desired_power=0.8, alpha=0.05):
    """Assess whether sample size is adequate for the observed effect."""
    
    # Compute achieved power
    achieved_power = compute_power(n=n, effect_size=effect_size, alpha=alpha)
    
    # Compute minimum N for desired power
    required_n = compute_sample_size(effect_size=effect_size, power=desired_power, alpha=alpha)
    
    return {
        'n_observed': n,
        'n_required_for_80_power': required_n,
        'achieved_power': achieved_power,
        'adequacy': 'adequate' if achieved_power >= desired_power else 'inadequate',
        'confidence_penalty': max(0, 1 - (n / required_n)) if n < required_n else 0,
        'recommendation': f"Need {required_n - n} more observations for {desired_power*100}% power"
            if n < required_n else "Sample size adequate"
    }
```

### 5.6 Measurement Error Propagation

```python
def propagate_measurement_error(correlation_result, measurement_errors):
    """Propagate measurement uncertainty through correlation estimates."""
    
    # Error-in-variables correction for attenuation
    reliability_m1 = 1 - (measurement_errors.m1**2 / correlation_result.variance_m1)
    reliability_m2 = 1 - (measurement_errors.m2**2 / correlation_result.variance_m2)
    
    # Disattenuated correlation (corrected for measurement error)
    r_observed = correlation_result.correlation
    r_corrected = r_observed / np.sqrt(reliability_m1 * reliability_m2)
    
    # Uncertainty in the correction
    correction_uncertainty = compute_correction_uncertainty(
        r_observed, reliability_m1, reliability_m2,
        n=correlation_result.n
    )
    
    return {
        'observed_correlation': r_observed,
        'error_corrected_correlation': min(r_corrected, 1.0),  # Cap at 1.0
        'correction_factor': 1 / np.sqrt(reliability_m1 * reliability_m2),
        'additional_uncertainty_from_correction': correction_uncertainty,
        'adjusted_confidence_interval': [
            r_corrected - 1.96 * correction_uncertainty,
            r_corrected + 1.96 * correction_uncertainty
        ]
    }
```

**Measurement Error Sources Tracked:**
- Instrument precision (lab analyzer CV)
- Device accuracy (wearable vs. medical-grade)
- Self-report bias (PRO instruments)
- Temporal misalignment (collection vs. result time)
- Missing data imputation uncertainty

### 6.3 Response Modeling

#### 6.3.1 Treatment Response Model

```python
class TreatmentResponseModel:
    """Model patient response to a treatment or intervention."""
    
    def __init__(self, intervention_date, pre_period='90D', post_period='90D'):
        self.intervention_date = intervention_date
        self.pre_period = pre_period
        self.post_period = post_period
    
    def fit(self, patient_timeline, outcome_variable):
        """Fit response model using interrupted time series."""
        
        # Pre-intervention trend
        pre_data = patient_timeline.slice(
            self.intervention_date - self.pre_period,
            self.intervention_date
        )
        pre_model = linear_trend_fit(pre_data[outcome_variable])
        
        # Post-intervention trend
        post_data = patient_timeline.slice(
            self.intervention_date,
            self.intervention_date + self.post_period
        )
        post_model = linear_trend_fit(post_data[outcome_variable])
        
        # Response components
        self.level_change = (
            post_model.intercept - pre_model.predict(self.intervention_date)
        )
        self.trend_change = post_model.slope - pre_model.slope
        self.response_latency = self.estimate_latency(patient_timeline, outcome_variable)
        
        # Uncertainty
        self.level_change_ci = bootstrap_ci(self.level_change, pre_data, post_data)
        self.trend_change_ci = bootstrap_ci(self.trend_change, pre_data, post_data)
        
        return {
            'level_change': (self.level_change, self.level_change_ci),
            'trend_change': (self.trend_change, self.trend_change_ci),
            'latency_days': self.response_latency,
            'model_fit_pre': pre_model.r_squared,
            'model_fit_post': post_model.r_squared
        }
    
    def classify_response(self):
        """Classify the nature of the response."""
        
        if self.level_change > 0 and self.trend_change >= 0:
            return 'POSITIVE_RESPONSE'  # Immediate and sustained improvement
        elif self.level_change > 0 and self.trend_change < 0:
            return 'FADING_RESPONSE'    # Initial improvement, now declining
        elif self.level_change <= 0 and self.trend_change > 0:
            return 'DELAYED_RESPONSE'   # Gradual improvement
        elif self.level_change < 0 and self.trend_change <= 0:
            return 'NEGATIVE_RESPONSE'  # Worsening
        else:
            return 'NO_CLEAR_RESPONSE'
```

| Rapid responder | Large positive | Stable positive | Strong immediate response |
| Slow responder | Small/neutral | Positive | Gradual improvement over time |
| Non-responder | Near zero | Near zero | No measurable effect |
| Initial responder | Large positive | Negative | Initial benefit, now losing effect |
| Adverse responder | Negative | Negative or worsening | Possible harm from treatment |

```python
class Nof1Trial:

            baseline_period='2W'
        )

            'intervention_B': intervention.control,

            'n_periods': n_periods,

            'measurement_schedule': measurement_schedule,
```

### 6.7 Interrupted Time Series (ITS)

ITS is a quasi-experimental design for evaluating the impact of an intervention:

```python
class InterruptedTimeSeries:
    """
    Interrupted Time Series analysis for intervention evaluation.
    
    Models:
    Y_t = β0 + β1*t + β2*I_t + β3*(t - T)*I_t + ε_t
    
    Where:
    - β0: Baseline level
    - β1: Baseline trend
    - β2: Immediate level change at intervention
    - β3: Trend change after intervention
    - I_t: Indicator (0 before, 1 after intervention)
    - T: Intervention time
    """
    
    def fit(self, time_series, intervention_time):
        """Fit ITS model."""
        
        # Create intervention indicator
        time_series['post'] = (time_series.index >= intervention_time).astype(int)
        time_series['time'] = (time_series.index - time_series.index[0]).days
        time_series['time_since_intervention'] = time_series['time'] * time_series['post']
        
        # Fit segmented regression
        model = sm.OLS(
            time_series['outcome'],
            sm.add_constant(time_series[['time', 'post', 'time_since_intervention']])
        )
        
        results = model.fit()
        
        return {
            'baseline_level': results.params['const'],
            'baseline_trend': results.params['time'],
            'level_change': results.params['post'],
            'trend_change': results.params['time_since_intervention'],
            'ci_95': results.conf_int(),
            'r_squared': results.rsquared,
            'residual_autocorrelation': durbin_watson(results.resid),
            'interpretation': self.interpret_its_results(results)
        }
```

```python
        # Build PubMed query
        pubmed_query = self.build_pubmed_query(entities)
        # e.g., "(atorvastatin[Title/Abstract]) AND (LDL[Title/Abstract]) 
        #        AND (hyperlipidemia[MeSH Terms])"
        
        # Execute search via E-utilities
        pubmed_results = self.search_pubmed(
            query=pubmed_query,
            filters={
                'publication_date': 'last_10_years',
                'article_type': ['randomized_controlled_trial', 'systematic_review', 
                                'meta_analysis', 'clinical_trial', 'guideline'],
                'humans_only': True,
                'english_only': True
            },
            retmax=max_results * 3  # Over-fetch for re-ranking
        )
        
        # Re-rank by semantic relevance to hypothesis
        hypothesis_embedding = self.embed(hypothesis.description)
        for article in pubmed_results:
            article.relevance_score = cosine_similarity(
                hypothesis_embedding,
                self.embed(article.abstract)
            )
        
        # Return top results with relevance scores
        return sorted(pubmed_results, key=lambda a: a.relevance_score, reverse=True)[:max_results]
```

```python
        for article in hypothesis.evidence.literature:

                'type': 'literature',
                'text': f"{article.authors[0]} et al. {article.title}. {article.journal}. {article.year}.",
                'pmid': article.pmid,
                'url': f"https://pubmed.ncbi.nlm.nih.gov/{article.pmid}/",
                'relevance_score': article.relevance_score
            })
        
        # CPIC citations
        for pgx in hypothesis.evidence.pharmacogenomics:
            citations.append({
                'type': 'cpic_guideline',
                'text': f"Clinical Pharmacogenetics Implementation Consortium. {pgx.gene}-{pgx.drug} Guideline. {pgx.guideline_date}.",
                'url': pgx.deep_link,
                'guideline_level': pgx.cpic_classification
            })
        
        # FDA label citations
        for label in hypothesis.evidence.fda_labels:
            citations.append({
                'type': 'fda_label',
                'text': f"FDA Prescribing Information: {label.medication}. Updated {label.version_date}.",
                'url': label.deep_link
            })
        
        return sorted(citations, key=lambda c: c.get('relevance_score', 1.0), reverse=True)
```

## 8. Clinician Interface

### 8.1 Overview

The clinician interface is the critical human-AI interaction layer. It must communicate complex multimodal findings clearly, prominently display uncertainty, enable efficient review, and never create the impression that DeepTwin's outputs are definitive clinical conclusions.

### 8.2 Design Principles

**CI-1: Transparency First**  
Every hypothesis shows its evidence, methods, and uncertainty. Nothing is presented as a black-box conclusion.

**CI-2: Clinician Control**  
Clinicians can override, dismiss, modify, or request additional analysis for any hypothesis. The system learns from these interactions.

**CI-3: Efficient Review**  
The interface supports rapid triage of multiple hypotheses with clear prioritization and at-a-glance indicators.

**CI-4: Context Preservation**  
Hypotheses are always shown in clinical context — patient demographics, current conditions, active medications — never in isolation.

**CI-5: Safety Prominence**  
Safety-critical indicators (exploratory labels, evidence grades, uncertainty warnings) are visually prominent and cannot be hidden.

## 9. Safety Architecture

### 9.1 Safety-First Design Philosophy

DeepTwin's safety architecture is built on the principle that **safety constraints must be architectural, not policy-based**. They cannot be bypassed through configuration, API parameters, or user actions. Safety is enforced in code, not in documentation.

### 9.2 Safety Principles

**SA-1: No Autonomous Clinical Decisions**  
DeepTwin never makes clinical decisions. It generates hypotheses for human review. There is no code path that translates a hypothesis directly into a clinical action (order, prescription, referral) without clinician approval.

**SA-2: Every Hypothesis Is Exploratory**  
Every output from DeepTwin is labeled as exploratory. The labels are generated automatically and cannot be removed. They appear in the UI, in API responses, and in all exports.

**SA-3: Evidence Grades on All Outputs**  
No output is generated without an evidence grade. The grading is mandatory and enforced by the hypothesis schema validator.

**SA-4: Population Context Included**  
Every hypothesis includes population context — how transferable is this finding to other patients with similar characteristics? This prevents over-generalization.

**SA-5: Uncertainty Prominently Displayed**  
Uncertainty is not hidden in metadata. It is displayed in the primary UI with visual indicators that are always visible.

**SA-6: Clinician Approval Required for Actions**  
Any action suggested by DeepTwin requires explicit clinician approval. The system does not implement "one-click actions" that bypass review.

## 11. API Design

### 11.1 API Overview

The DeepTwin API follows RESTful principles with JSON payloads. All endpoints require authentication and enforce role-based access control. Every response includes safety metadata that cannot be suppressed.

## Appendix A: Glossary of Terms

| Term | Definition |
|---|---|
| **Multimodal** | Involving multiple types or modes of data (lab, vital signs, genetics, etc.) |
| **Correlation** | A statistical relationship between two variables — does not imply causation |
| **Hypothesis** | A proposed explanation for an observation, requiring testing and validation |
| **Confidence Interval (CI)** | A range of values that likely contains the true parameter value |
| **Evidence Grade** | A rating (A/B/C/D) indicating the strength of evidence supporting a finding |
| **Temporal Lag** | The time delay between a cause and its observed effect |
| **Granger Causality** | A statistical test of predictive precedence — not true causation |
| **Dynamic Time Warping** | A method for measuring similarity between temporal sequences |
| **Bootstrap** | A resampling method for estimating confidence intervals |
| **N-of-1 Trial** | A single-patient clinical trial comparing treatments |
| **CPIC** | Clinical Pharmacogenetics Implementation Consortium |
| **PGx** | Pharmacogenomics — study of how genes affect drug response |
| **PROMIS** | Patient-Reported Outcomes Measurement Information System |
| **SDOH** | Social Determinants of Health |
| **Exploratory** | Intended for investigation and hypothesis generation, not definitive conclusions |

## Appendix C: Algorithm Selection Matrix

| Clinical Question | Recommended Methods | Avoid | Rationale |
|---|---|---|---|
| Med → Lab relationship | Pearson/Spearman with lag, MI | Granger alone | Simple temporal association |
| Symptom → Vital relationship | DTW, lag correlation | Granger (too sparse) | Irregular timing, shape matters |
| Multi-panel Lab changes | CCA | Multiple pairwise | Reduce dimensionality |
| Treatment response | ITS, Bayesian trend | Simple pre/post | Account for trend |
| Adverse event detection | Anomaly detection + temporal context | Any single method | Safety-critical, need convergence |
| Genotype → Phenotype | Association testing, CPIC lookup | Correlation | Genetic relationships need specific methods |

## Appendix D: Regulatory Considerations

### D.1 FDA Perspective

DeepTwin is designed as a **Clinical Decision Support (CDS)** tool under the FDA's CDS guidance framework. Key design decisions supporting this classification:

- **Intended to support, not replace:** Clinician review is mandatory for all outputs
- **Transparent:** Evidence, methods, and uncertainty are displayed
- **Clinician can independently review:** All underlying data is accessible
- **Does not automate diagnosis or treatment:** No autonomous clinical decisions
